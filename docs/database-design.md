# 数据库设计文档

## 1. 设计目标

本数据库用于支持「思维导图式记事本」的核心数据存储需求，
包括导图元数据、节点结构以及节点之间的连接关系。

设计目标：
- 支持思维导图的创建、编辑、保存与加载
- 支持节点树结构与自由布局
- 支持节点折叠 / 展开
- 支持节点之间的连接关系
- 支持后续多导图扩展
- 使用 SQLite 作为本地持久化存储

---

## 2. 表结构总览

| 表名 | 说明 |
|-----|-----|
| mind_map | 思维导图元数据表 |
| node | 思维导图节点表 |

## 4. 设计说明

（用于说明表之间的关系、设计取舍与约束）

### 4.1 表之间的关系说明

本系统围绕“思维导图”这一核心业务对象，采用 mind_map / node / edge 三表结构进行建模，各表之间关系如下：

- mind_map（导图表）

  - 表示一张完整的思维导图

  - 是 node 与 edge 的逻辑归属单位

- node（节点表）

  - 表示导图中的一个节点

  - 通过 mind_map_id 归属于某一张导图

  - 通过 parent_node_id 构成树形父子结构

- edge（连接表）

  - 表示节点之间的连接关系

  - 通过 source_node_id 与 target_node_id 连接两个节点

  - 通过 mind_map_id 约束连接关系仅存在于同一张导图中

#### 三张表之间的逻辑关系如下：

- 一张 mind_map 可以包含多个 node

- 一张 mind_map 可以包含多条 edge

- 一个 node 只能属于一张 mind_map

- 一个 edge 只能连接同一张 mind_map 内的两个 node

### 4.2 父子关系与连接关系的设计取舍

在本设计中，节点的父子关系与节点之间的连接关系被刻意拆分，原因如下：

#### 父子关系（树结构）

通过 node.parent_node_id 实现

**用于表达：**

思维导图的层级结构

节点折叠 / 展开逻辑

**优点：**

树结构清晰

查询子节点高效

折叠逻辑实现简单

#### 连接关系（图结构）

通过 edge 表实现

**用于表达：**

节点之间的可视化连线

非严格树形的扩展连接（如跨分支引用）

**优点：**

支持非树状结构扩展

不破坏原有父子层级模型

该设计同时兼顾 树结构的稳定性 与 图结构的扩展性，是思维导图类应用中较为通用且可靠的建模方式。

### 4.3 节点位置与布局的设计说明

- 节点的位置信息通过 node.x、node.y 字段存储

- 坐标值由前端负责计算与更新，后端仅负责持久化

- 布局方向（如 LEFT_RIGHT、TOP_DOWN）为导图级属性，存储于 mind_map.layout_direction

**该设计的优势在于：**

- 后端不耦合具体布局算法

- 前端可自由切换或重算布局

- 数据模型保持稳定

### 4.4 状态字段的归属原则

在字段设计过程中，遵循以下原则：

- 节点自身状态 → node 表 如：collapsed（是否折叠）

- 导图全局状态 → mind_map 表 如：layout_direction

- 连线关系 → edge 表

- 不承担节点结构含义

- 通过明确状态归属，避免字段语义混乱，提高模型可维护性。

### 4.5 时间字段设计说明

所有核心业务表均包含时间字段：

- created_at：记录数据创建时间

- updated_at：记录最近一次修改时间（mind_map、node）

**设计目的：**

- 支持数据变更追踪

- 支持后续版本控制或自动保存机制

- 便于调试与问题定位

### 4.6 约束与一致性说明

- 所有 node 与 edge 必须通过 mind_map_id 归属于某一张导图

- edge.source_node_id 与 edge.target_node_id 必须指向同一导图内的节点（逻辑约束）

- 根节点的 parent_node_id 为 NULL

- 本阶段不启用软删除（deleted 字段），以降低模型复杂度

### 4.7 扩展性设计说明

当前数据库设计已为以下扩展场景预留空间：

- 多导图 / 多画布支持

- 非树状节点连接

- 节点样式、连线样式扩展（可通过新增字段或附加表实现）

- 权限、多用户协作（可在 mind_map 层扩展）

- 在不破坏现有表结构的前提下，可通过新增字段或关联表实现功能增强。
